\# Библия Проекта: Viral Engine by Sherlock

\## Документ 06: Безопасность, Риски и Масштабирование



\### 1. Отказоустойчивость API (Failover Strategy)

Система не должна зависеть от одного поставщика:

\* \*\*Видео:\*\* Если API Luma лежит, система должна иметь "запасной путь" на Runway или Kling.

\* \*\*Аватар:\*\* Если HeyGen перегружен, мы уведомляем пользователя о задержке, но не отменяем проект.

\* \*\*Очередь:\*\* Использование Redis (Upstash) для управления очередью задач, чтобы рендер-воркер не "захлебнулся".



\### 2. Политика безопасности данных

\* \*\*Zero-Retention:\*\* Мы официально декларируем, что не храним исходные селфи-видео пользователей после создания Instant Avatar. Это критично для доверия (GDPR compliance).

\* \*\*Encryption:\*\* Все API-ключи пользователей (в режиме Pro) хранятся в зашифрованном виде в Vault Supabase.



\### 3. Борьба с абузом (Abuse Prevention)

\* \*\*Приветственные кредиты:\*\* Выдаются только после верификации Telegram-аккаунта (который должен быть старше 3 месяцев).

\* \*\*Контент-фильтр:\*\* Интеграция модерации Gemini. Если пользователь пытается сгенерировать запрещенный контент (политика, насилие, adult), система блокирует генерацию без возврата кредитов.



\### 4. План технической поддержки

\* \*\*L1 Support (Бот):\*\* База знаний внутри Telegram-бота.

\* \*\*L2 Support (Human):\*\* Тот самый администратор-Штирлиц, который имеет доступ к админ-панели для ручного начисления кредитов в случае сбоя API.
